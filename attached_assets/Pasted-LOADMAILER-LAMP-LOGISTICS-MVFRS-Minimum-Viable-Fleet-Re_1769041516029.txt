LOADMAILER / LAMP LOGISTICS — MVFRS (Minimum Viable Fleet Reliability System)
Maintenance + Inspections + Compliance inside your existing Replit trucking app

Goal:
1) Prevent breakdowns (PM discipline + early defect capture)
2) Make inspections feed maintenance automatically (defect → work order)
3) Create accountability (driver + dispatcher + maintenance coordinator)
4) Provide a single system of record (your app), while integrating best-in-class tools (QB, ELD, e-sign)

- “Trucks Out of Service” list
- “Open Work Orders by Priority” (CRITICAL/URGENT/ROUTINE)

- Truck list (status, odometer, next PM due, safety hold)
- Truck detail:
  - Info tab (VIN/unit/make/model)
  - Maintenance tab (PM schedule + completed work orders)
  - Inspections tab (DVIR history + fleet inspections)
  - Documents tab (insurance/registration/annual insp)
  - Vendors tab (preferred vendors by zip/state)

  - choose truck
  - choose type (pre/post/weekly/monthly)

  - checklist with OK / Needs Attention
  - if Needs Attention → require severity + notes + photo for critical categories

- Kanban style: OPEN → TRIAGED → ASSIGNED → IN_PROGRESS → COMPLETED → CLOSED
- Work order detail:
  - priority + safety hold
  - vendor assignment

  - close-out summary

5) Vendors
- directory with category filters
- quick-call buttons
- “Preferred vendors by region” (TN/GA/AL/NC)

6) Compliance / Documents
- expiring next 60/30/7 days
- upload/replace doc
- document history

Optional (later):
7) Analytics
- cost per mile by truck
- downtime hours by truck
- recurring failure categories
- “sell/replace flags” dashboard

C) AUTOMATIONS (THE HIGH-ROI RULES)
These are the automations that make your system behave like a fleet operator.

Use your existing NextStep webhook for push notifications:
POST https://nextstephq.app/api/webhook
Headers: Content-Type: application/json
Body: { "message": "...", "title": "...", "priority": 8 }

1) AUTOMATION: DEFECT → WORK ORDER AUTO-CREATE
Trigger: inspection_items.status = NEEDS_ATTENTION
Logic:
- Create work_order with:
  - source = INSPECTION
  - priority = severity mapped:
      CRITICAL → CRITICAL + safety_hold=true + truck.status=OUT_OF_SERVICE
      URGENT   → URGENT
      ROUTINE  → ROUTINE
  - issue_category inferred from item_code (TIRES/BRAKES/etc.)
  - symptoms = defect_notes
  - related_inspection_id = inspection.id
Notify:
- If CRITICAL:
  - Push to OWNER + DISPATCHER + MAINT_COORD
  - Title: “CRITICAL DEFECT: Truck {unit_number}”
  - Message: “Truck is on SAFETY HOLD. Work Order {id} created. Resolve before dispatch.”
- If URGENT:
  - Push to DISPATCHER + MAINT_COORD
- If ROUTINE:
  - In-app only or daily digest

2) AUTOMATION: PM DUE SOON / DUE / OVERDUE
Trigger: daily scheduler at 6:30 AM local (America/New_York)
Logic:
- For each truck:
  - if due within grace window → DUE_SOON
  - if at/over threshold → DUE
  - if beyond grace → OVERDUE + optionally safety_hold=true (your policy)
Notify:
- DUE_SOON (500 miles / 7 days):
  - Push DISPATCHER + MAINT_COORD
- DUE:
  - Push DISPATCHER + MAINT_COORD + OWNER
- OVERDUE:
  - Push OWNER (priority 9) + set truck “Dispatch Block” flag (recommended)

3) AUTOMATION: “BREAKDOWN MODE” BUTTON
Trigger: create breakdown_report
Logic:
- Auto-create work_order with source=BREAKDOWN priority=CRITICAL
- Set truck.status=OUT_OF_SERVICE and safety_hold=true
- Start downtime timer (downtime_start_at=now)
Notify:
- OWNER + DISPATCHER + MAINT_COORD
Message includes:
- location_text
- can_move yes/no
- hazard yes/no
- vendor category suggestions (TOWING vs MOBILE_MECHANIC)
Add quick action checklist in work_order_events:
- “Notify broker/customer using template”
- “Assign vendor”
- “Update ETA”
- “Close breakdown”

4) AUTOMATION: DOCUMENT EXPIRY COUNTDOWN (60/30/7)
Trigger: daily scheduler at 7:00 AM
Logic:
- Find documents expiring in 60/30/7 days
Notify:
- 60 days: accounting/ops
- 30 days: ops + owner
- 7 days: owner priority 9
Message:
- “DOC EXPIRING: {doc_type} for {truck/unit or driver} on {date}”

5) AUTOMATION: ODOMETER UPDATE REQUIRED
Trigger: driver submits inspection without odometer OR truck current_odometer stale > 7 days
Notify:
- DRIVER + DISPATCHER (if assigned)
Message:
- “Odometer update required for Truck {unit}. Enter latest miles before next dispatch.”

6) AUTOMATION: CLOSE-OUT QUALITY GATE (forces discipline)
Trigger: attempting to set work_order status COMPLETED/CLOSED
Require:
- actual_cost entered (or set 0 with reason)
- resolution_notes entered
- downtime_end_at set if downtime started
This is how you build real analytics and prevent “messy history.”

D) ROLE-BASED PERMISSIONS (NON-NEGOTIABLE)
OWNER / ADMIN:
- Full access, overrides, policy settings, analytics

DISPATCHER:
- View trucks, create work orders, view/triage, cannot delete
- Can place truck “temporary hold” but cannot remove safety hold without MAINT/OWNER

MAINT_COORD:
- Work orders full, vendor assignment, costs, close-out
- Can remove safety hold after resolution + notes

DRIVER:
- Submit inspections + breakdown reports
- View their own assigned truck status + required tasks
- Cannot see cost fields

ACCOUNTING:
- View costs + exports + documents
- Read-only on inspections
